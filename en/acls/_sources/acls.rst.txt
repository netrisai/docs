##########################
Access Control Lists (ACL)
##########################
Netris supports ACL-based network access control on switches. Each entry you configure — a Netris ACE (Access Control Entry) — matches traffic by source and destination IP address, port or port range, and protocol, and applies a Permit or Deny action.

.. image:: images/acl-main-view.png
    :align: center
    :class: with-shadow

.. raw:: html

   <p style="text-align: center;"><em>Figure: Netris ACL main view</em></p>

ACEs execute in switch hardware, providing line-rate performance for security enforcement. Because Netris only installs an ACE on the switches where it's actually needed (see "Where ACEs Are Installed"), there's no fixed limit on the number of ACEs you can create in Netris — but the number that can be installed on any single switch is limited by that switch's TCAM capacity.

.. image:: images/acl-tcam.png
    :align: center
    :class: with-shadow

.. raw:: html

   <p style="text-align: center;"><em>Figure: TCAM utilization view under Net→Inventory</em></p>

Netris applies several optimization algorithms to minimize TCAM usage while meeting the user-defined requirements.

ACL Default Policy
-------------------
The ACL default policy is to permit all hosts to communicate with each other. You can change the default policy on a per-site basis by editing the Site features under Net→Sites. Once the "ACL Default Policy" is changed to "Deny," the specified site will start dropping traffic unless specific communication is permitted by an ACE.

.. image:: images/acl_site_default.png
    :align: center
    :class: with-shadow

.. raw:: html

   <p style="text-align: center;"><em>Figure: Changing the ACL Default Policy for a site</em></p>

ACL Rules
---------
ACEs can be created, listed, and edited under Services→ACL. IPv4 and IPv6 addresses can't be mixed within the same entry — each ACE matches one address family.

.. image:: images/acl_action_permit.png
    :align: center
    :class: with-shadow

.. raw:: html

   <p style="text-align: center;"><em>Figure: Example ACL rule permitting SSH traffic with Established</em></p>

Description of ACE fields.
==========================

* **Name** - Unique name for the ACL entry.
* **VPC** - The VPC Netris uses to look up the route to the rule's source address (destination address, for the reverse rule generated by Established/Reverse) when deciding where to install the ACE. See "Where ACEs Are Installed" below.
* **Protocol** - IP protocol to match.

  * All - Any IP protocols.
  * IP - Specific IP protocol number.
  * TCP - TCP.
  * UDP - UDP.
  * ICMP ALL - Any IPv4 ICMP protocol.
  * ICMP Custom - Custom IPv4 ICMP type. See "ICMP Custom values" below.
  * ICMPv6 ALL - Any IPv6 ICMP protocol.
  * ICMPv6 Custom - Custom IPv6 ICMP code.

* **Active Until** - Disable this rule at the defined date/time.
* **Action** - Permit or Deny forwarding of matched packets.
* **Established/Reverse** - For TCP, also match reverse packets except with TCP SYN flag. For non-TCP, also generate a reverse rule with the source/destination swapped.

**ICMP Custom values** (Protocol = ICMP Custom) - each value sets the ACE to match one ICMPv4 type. Several of these are deprecated and unlikely to appear in real traffic, but Netris still lets you match on them.

**Source/Destination** - Source and destination addresses and ports to match.

* **Source** IPv4/IPv6 - IPv4/IPv6 address.
* **Ports Type**

  * Port Range - Match on the port or a port range defined in this window.
  * Port Group - Match on a group of ports defined under Services→ACL Port Group.

* **From Port** - Port range starting from.
* **To Port** - Port range ending with.
* **Comment** - Descriptive comment.
* **Check button** - Checks the ACE you're adding, editing, or copying against existing ACEs, and flags any existing ACE whose matched traffic overlaps with it — whether that ACE is fully shadowed by (entirely contained within) the candidate, or only partially overlaps with it.

.. warning::

    At least one of Source or Destination must fall within an existing IPAM entry of type Subnet (not Allocation) — Netris uses this to confirm the ACE is anchored to your deployment. The address doesn't have to match a Subnet entry exactly; it just needs to fit entirely within one; for example, source 10.0.0.1/32 is valid as long as an IPAM Subnet of 10.0.0.0/24 (or any Subnet-type entry that contains it) exists, even though 10.0.0.1/32 itself was never entered into IPAM. If neither Source nor Destination fits within a known Subnet, Netris rejects the ACE with "Both Source and Destination addresses are out of this deployment."

Checking for overlapping rules
------------------------------
.. image:: images/acl-check-button.png
    :align: center
    :class: with-shadow

.. raw:: html

   <p style="text-align: center;"><em>Figure: Check button indicates other rules with partial match to the candidate ACE</em></p>

Example: Checking a candidate ACE for TCP from 10.0.0.0/8 to 10.1.0.0/24 surfaces two existing ACEs: TCP from 10.0.0.0/24 to 10.1.0.0/24 (fully shadowed — that traffic is entirely contained within the candidate's), and TCP from 10.1.0.0/24 to 10.0.0.0/8 (a partial match — since 10.1.0.0/24 is a subset of 10.0.0.0/8, any traffic from 10.1.0.0/24 to 10.1.0.0/24 satisfies both rules' address ranges, even though neither rule is a superset of the other).

ACL Processing Order
--------------------
Netris doesn't organize entries into multiple, separately named ACLs — every entry you create is one ACE, evaluated together in a single ordered list. Netris determines the evaluation order across your ACEs automatically; there is no user- or API-exposed priority field. ACEs are sorted as follows:

#. Sort by source IPv4/IPv6 prefix, longest to shortest.
#. For rules with the same source prefix, sort by destination prefix, longest to shortest.
#. For rules with the same source and destination prefix, sort by protocol, in the order: ICMP, UDP, TCP, ANYTHING_ELSE, IP.
#. For rules with the same source prefix, destination prefix, and protocol, sort by source port range, smallest to largest.
#. For rules with the same source prefix, destination prefix, protocol, and source port range, sort by destination port range, smallest to largest.

Evaluation stops at the first rule that matches a given packet — that rule's action (Permit or Deny) applies, and no further rules are checked. Action isn't itself part of the sort order: a more specific Permit rule can take effect ahead of a broader Deny rule if it matches first, and vice versa. This gives more specific rules precedence over general ones, similar to longest-prefix-match behavior in routing.

A rule with "Established/Reverse" enabled generates a second, independent rule with source and destination swapped. That reverse rule is sorted into the list by the same criteria above, and — because installation location is based on a route lookup on each rule's own source address (see "Where ACEs Are Installed" below) — it's often installed on a completely different switch, or SoftGate, than its forward rule, not merely sorted differently within the same list.

After all user-defined rules, Netris appends one implicit rule matching the site's ACL Default Policy: ``deny ip any any`` if the policy is Deny, or ``permit ip any any`` if the policy is Permit.

Because a single ACE is always one address family or the other (see above), this sort order never has to resolve precedence between IPv4 and IPv6 entries.

ANYTHING_ELSE in the protocol order covers protocols that carry TCP flags; these flags aren't directly configurable on a rule, but checking "Established" automatically sets the appropriate flags on the generated reverse rule. IP, by contrast, refers to selecting a specific IP protocol number in the Netris controller — choose "All" to match every protocol.

Where ACEs Are Installed
------------------------
You don't choose where an ACE is enforced — Netris determines that automatically from your fabric's topology and the ACE's VPC field. On the switch fabric, Netris installs the ACE on whichever switch a route lookup shows is closest to the source — specifically, on the Layer-3 interface closest to the source. When traffic reaches a VPC through a SoftGate, the ACE is installed on the SoftGate's output chain.

Because every Layer-3 interface belongs to exactly one VPC, two VPCs with overlapping or identical IP ranges never conflict, even if both have sources on the same physical switch: the route lookup happens in the VPC selected on the ACE, and the resulting ACE always lands on an L3 interface belonging to that VPC.
